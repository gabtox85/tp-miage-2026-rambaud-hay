*Le code est-il structuré en couches ? Quel problèmes ce code peut-il poser ?*

Cela pose plusieurs problèmes :
- Fort couplage entre les composants et non-respect du principe de responsabilité unique.

*Où se trouve le code métier (voir la règle de gestion RG 1) ?*

Les règles de gestion (RG 1) se trouvent directement mélangées au code du contrôleur REST. Elles sont implémentées dans la méthode privée `finalContent` de la classe `TodoListController.java`.

*Cette règle est-elle facilement testable par un test unitaire ?*

Non, cette règle n'est pas facilement testable unitairement :
- La méthode `finalContent` qui l'implémente est privée : pour la tester, il faut passer par la méthode publique `todoItems()`, ce qui implique de devoir mocker le comportement de `todoItemRepository` (la base de données) pour pouvoir tester de la simple logique métier.
- La règle fait appel à `Instant.now()`. L'heure du système n'étant pas injectée ou contrôlée de l'extérieur, la méthode exécutée dépend de l'heure courante d'exécution, ce qui rend l'assertion des tests instable.


* Lancer une base PostgreSQL en Docker dans un terminal (on lance ici la base en mode interactif pour visualiser son activité. Pour la lancer en tâche de fond, remplacer les options `it` par `d` comme 'daemon'):
```bash
docker run -it -e POSTGRES_PASSWORD=password -p 5432:5432 postgres
```
*Expliquer cette ligne de commande (y compris les options utilisées)*

- `docker run` : Commande principale pour créer et démarrer un nouveau conteneur à partir d'une image Docker.
- `-it` : Exécute le conteneur en mode interactif (`-i`) et alloue un pseudo-terminal (`-t`). Cela permet d'attacher le terminal actuel pour voir l'activité et les logs de PostgreSQL.
- `-e POSTGRES_PASSWORD=password` : Définit une variable d'environnement (`-e`). Ici, elle initialise le mot de passe du superutilisateur PostgreSQL.
- `-p 5432:5432` : Redirige les ports (`-p`). Mappe le port `5432` de la machine hôte vers le port `5432` du conteneur, rendant la base accessible depuis l'extérieur du conteneur.
- `postgres` : Le nom de l'image Docker officielle à exécuter.

* Compléter le code manquant dans la méthode `TodoListController.createTodoItem()`

*Pourquoi `todoItemRepository` est-il `null` ? Quelle est la meilleure façon de l'injecter ?*

`todoItemRepository` est `null` car Spring utilise par défaut le constructeur sans argument pour instancier le contrôleur. L'autre constructeur, qui assigne l'attribut, n'est pas appelé.
La meilleure façon de l'injecter est d'utiliser l'**injection par constructeur**. Pour cela, la solution est de **supprimer le constructeur vide**. Ainsi, Spring sera obligé d'utiliser le seul constructeur disponible et d'injecter automatiquement l'instance de `TodoItemRepository` correspondante.

### Exercice 2 - Refactoring en architecture hexagonale
_Temps estimé : 1 h 20_

* Partir du projet `todolist-debut-ex2`

NOTE: Le projet a été réusiné suivant les principes de l'architecture hexagonale : 

image::images/archi_hexagonale.png[]
Source : http://leanpub.com/get-your-hands-dirty-on-clean-architecture[Tom Hombergs]

* Nous avons découpé le coeur en deux couches : 
  - la couche `application` qui contient tous les contrats : ports (interfaces) et les implémentations des ports d'entrée (ou "use case") et qui servent à orchestrer les entités.
  - la couche `domain` qui contient les entités (au sens DDD, pas au sens JPA). En général des classes complexes (méthodes riches, relations entre les entités)

*Rappeler en quelques lignes les grands principes de l'architecture hexagonale.*

L'architecture hexagonale, ce repose sur 3 éléments importants pour que le code reste propre :

1. **Séparer explicitement User-Side, Business Logic et Server-Side** : on mélange pas les torchons et les serviettes. L'interface utilisateur , la logique métier, et le côté serveur ont chacun leur espace pour éviter de faire un gros plat de spaghettis.
2. **Les dépendances vont vers la Logique Métier** : Tout ce qui l'entoure doit dépendre de lui pour s'adapter, mais le métier se fiche totalement de la technique, il dépend de personne.
3. **On isole les frontières par des Ports et Adapters** : Pour brancher le monde extérieur au coeur métier, on plug des prises (les *Ports*, les interfaces Java) et des embouts (les *Adapters*). Si on change de composants technique il y auras juste besoin de changer les embouts.

Compléter ce code avec une fonctionnalité de création de `TodoItem`  persisté en base et appelé depuis un endpoint REST `POST /todos` qui :

* prend un `TodoItem` au format JSON dans le body (voir exemple de contenu plus haut);
* renvoie un code `201` en cas de succès. 

La fonctionnalité à implémenter est contractualisée par le port d'entrée `AddTodoItem`.
